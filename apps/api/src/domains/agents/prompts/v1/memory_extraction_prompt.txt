Extract NEW user facts from the LAST user message into long-term memory.
Return [] if nothing is worth memorizing.

## WHAT TO EXTRACT (ALL four must hold)
- **Durable utility** — helps personalize future unrelated conversations
- **Stable** — lasts weeks+ (not a transient mood, state, or task)
- **Unique** — not derivable from user profile, settings, or system context
- **Actionable** — enables meaningful tone or content adaptation

## WHAT NOT TO EXTRACT
- Transient states ("I'm tired today", "busy right now")
- Conversational feedback ("thanks", "shorter please", "perfect")
- Task-only details (one-shot recipients, scheduled event specifics)
- Facts already in EXISTING MEMORIES with equivalent content

## RULES (all mandatory)

1. **First person** — Write as the user speaks: "I like X", "My son is Y". NEVER "The user...". Keep correct subject ("my husband sings" ≠ "I sing").
2. **Facts exact, affect interpretable** — Quote facts as stated (names, dates, numbers, places). You MAY infer emotional intensity from language ("I hate", "breaks me", flat "ok"). NEVER diagnose ("anxious", "introvert") or invent facts.
3. **Atomic** — One fact per memory. Decompose composite statements. "My brother Jean got married last year" → 2 memories: a `relationship` (Jean) + an `event` (wedding, dated).
4. **Full names** — If KNOWN RELATIONSHIPS provides the full name, use it ("my son" → "My son Martin Dupond").
5. **Absolute dates** — Resolve all temporal references using CURRENT DATE AND TIME. Never store "tomorrow", "last week", "soon". Use "on 2026-04-12", "in April 2026", "week of 2026-04-13".
6. **Semantic recallability** — Write content to match how the user or assistant may later refer to it. Favor generic vocabulary over slang. Include the concept AND the specifics ("My daughter Emma, 8, loves drawing" > "Emma draws"). Pick `trigger_topic` = canonical concept, not just a name ("family daughter Emma" > "Emma").
7. **Content language** — Write `content`, `trigger_topic`, `usage_nuance` in the user's message language. Keep JSON keys, `action`, and `category` values in English.

## CATEGORIES (choose the narrowest fit; decompose when multiple fit)
- `preference` — enduring likes/dislikes/tastes
- `personal` — stable identity attribute (job, city, nationality, belief)
- `relationship` — a person + their link to the user (include name if known)
- `event` — a datable one-time occurrence, past or future (with resolved date)
- `pattern` — recurring behavior or habit over time ("I run every morning")
- `sensitivity` — pain, trauma, illness, grief, conflict, active avoidance

## QUALIFICATION

**emotional_weight** (−10 trauma → +10 joy):
- 0: factual, no affect ("I work as an accountant")
- ±1 to ±2: light affect ("I'm not a fan of X")
- ±3 to ±6: marked ("I love", "it hurts")
- ±7 to ±10: intense ("devastating", "lost my mother")

**importance** (0.0 → 1.0, avoid default 0.7 by laziness):
- 0.3 minor preference · 0.5 standard context · 0.7 meaningful relationship or stable identity · 0.9 core identity, major event, sensitivity

**usage_nuance** (≤300 chars) — concrete directive to the future assistant:
- `sensitivity` category OR `emotional_weight ≤ −3`: frame as OBLIGATION ("Never joke about...", "Do not bring up unless user does")
- Otherwise: frame as hint ("Use when discussing...", "Helpful for personalizing...")

**trigger_topic** (≤100 chars) — canonical concept-level keyword matching likely future queries, not just a name.

## ACTIONS

JSON array only. No markdown, no comments. Use ONLY UUIDs from EXISTING MEMORIES. Never invent IDs. Memories tagged `[PINNED]` are user-locked: NEVER emit update or delete for them (if a new fact contradicts a pinned memory, create a new memory with the new fact — the two will coexist).

**create** — new fact, no equivalent existing memory:
```json
{{"action":"create","content":"...","category":"...","emotional_weight":0,"trigger_topic":"...","usage_nuance":"...","importance":0.5}}
```

**update** — actively check: if any existing memory refers to the SAME entity (person, place, job, attribute) and the new fact changes/enriches/contradicts it → emit update. Rewrite the full content.
```json
{{"action":"update","memory_id":"<UUID>","content":"...","emotional_weight":0}}
```

**delete** — existing memory is now obsolete or was misrecorded and has no replacement.
```json
{{"action":"delete","memory_id":"<UUID>"}}
```
