FOR_EACH REQUIREMENT (MANDATORY — from query analysis):

The user wants the action applied to EACH {collection} INDIVIDUALLY.
Expected cardinality: {cardinality_hint}

MANDATORY 2-STEP STRUCTURE (DO NOT collapse into 1 step):
Step 1 = RETRIEVE: get_{collection}_tool → produces $steps.step_1.{collection}
Step 2 = ACTION: mutation tool with for_each="$steps.step_1.{collection}", for_each_max={for_each_max_value}

CRITICAL: A step CANNOT reference its own output. for_each ALWAYS points to a PREVIOUS step.
WRONG: {{"id":"step_1","tool_name":"delete_email_tool","for_each":"$steps.step_1.emails"}} ← self-reference!
RIGHT: {{"id":"step_1","tool_name":"get_emails_tool"}}, {{"id":"step_2","tool_name":"delete_email_tool","for_each":"$steps.step_1.emails","parameters":{{"message_id":"$item.id"}}}}

SCHEMA CONSTRAINT: for_each_max MUST be {for_each_max_value} (hard validation limit, system handles pagination).

$item SYNTAX (CRITICAL — use EXACTLY as written):
- CORRECT: "$item.email", "$item.summary", "$item.start.dateTime"
- WRONG: "$steps.step_1.{collection}[0].field" → only first item!
- WRONG: "$steps.step_1.{collection}[*].field" → wildcard invalid!
- WRONG: any [INDEX], [i], [N] in for_each step parameters

PARAMETER BINDING: For each parameter, ask "should this differ per item?"
YES → "$item.field" | NO → static value.

CROSS-DOMAIN: The action tool (step 2) CAN be from a DIFFERENT domain than the collection.

RELATIVE DATETIME: When user wants timing relative to each item ("before each", "day before"):
Use `relative_trigger` if supported: "$item.DATETIME_FIELD|OFFSET|@TIME" (OFFSET: [+-]Nd/h/m).
